# Техническое задание для мобильного приложения

## 1. Цель

Разработать мобильное приложение для учёта магазина, в котором пользователь может быстро:

- посмотреть текущее состояние магазина;
- оформить продажу;
- работать с товарами и остатками;
- контролировать долги;
- добавлять расходы;
- просматривать базовую аналитику.

Приложение должно быть простым, быстрым и ориентированным на ежедневную работу продавца и владельца магазина.

## 2. Основные роли

### 2.1. Seller

Должен иметь доступ к:

- главной странице;
- товарам;
- продажам;
- долгам;
- своему профилю.

Ограничения:

- без управления магазином;
- без управления пользователями;
- без административных настроек.

### 2.2. Owner

Должен иметь доступ ко всему, что есть у seller, а также к:

- расходам;
- закупкам;
- пользователям магазина;
- отчётам;
- настройкам магазина.

### 2.3. Super Admin

Для `super_admin` основной рабочий интерфейс должен оставаться в web/admin части.
В мобильном приложении можно ограничиться базовым доступом или не делать отдельный мобильный сценарий.

## 3. Нижняя навигация

Рекомендуемый состав нижнего меню:

- `Главная`
- `Товары`
- `Продажи`
- `Долги`
- `Ещё`

В раздел `Ещё` выносятся:

- расходы;
- закупки;
- отчёты;
- настройки;
- пользователи;
- профиль;
- выход.

## 4. Главная страница

Главная страница должна быть самым полезным экраном приложения.

### 4.1. Блоки на экране

#### Верхняя часть

- название магазина;
- приветствие;
- дата;
- при необходимости текущая смена.

#### Основные KPI-карточки

- продажи за сегодня;
- расходы за сегодня;
- прибыль;
- общая сумма долгов.

#### Быстрые действия

- `Новая продажа`
- `Добавить товар`
- `Добавить расход`
- `Добавить долг`

#### Важные предупреждения

- товары с низким остатком;
- неоплаченные долги;
- при необходимости предупреждение о низкой активности.

#### Последние события

- последние продажи;
- последние расходы;
- последние операции по долгам.

### 4.2. Что должно приходить с backend

Для удобства мобильного приложения желательно сделать агрегирующий endpoint:

- `GET /api/v1/dashboard`

Пример состава ответа:

- `today_sales_total`
- `today_expenses_total`
- `today_profit`
- `debts_total`
- `low_stock_count`
- `recent_sales`
- `recent_expenses`
- `low_stock_products`

## 5. Раздел «Товары»

Это один из ключевых разделов приложения.

### 5.1. Экран списка товаров

Должен содержать:

- поиск;
- фильтр:
  - все;
  - мало товара;
  - нет в наличии;
- список карточек товаров.

Карточка товара должна показывать:

- фото товара;
- название;
- код;
- остаток;
- единицу измерения;
- цену продажи;
- индикатор остатка.

### 5.2. Экран карточки товара

Должен содержать:

- фото товара;
- название;
- код;
- цена закупки;
- цена продажи;
- текущий остаток;
- минимальный остаток;
- действия:
  - `Изменить`;
  - `Добавить закупку`;
  - `Посмотреть движение`.

### 5.3. Экран создания / редактирования товара

Поля:

- фото;
- название;
- код;
- единица измерения;
- цена закупки;
- цена продажи;
- стартовый остаток;
- минимальный остаток.

### 5.4. Backend

Сейчас основа уже есть:

- список товаров;
- просмотр товара;
- создание товара;
- обновление товара;
- удаление товара;
- поддержка картинки товара.

Желательно дополнительно возвращать:

- `is_low_stock`;
- `image_url`.

## 6. Движение товара

Это отдельный важный экран.

### 6.1. Назначение

Пользователь должен видеть, как товар приходил и как уходил.

### 6.2. Экран должен содержать

- шапку товара;
- текущий остаток;
- список движений по дате.

### 6.3. Каждая запись движения должна содержать

- дату;
- тип операции:
  - закупка;
  - продажа;
  - позже можно добавить корректировку;
- количество;
- цену;
- сумму;
- кто выполнил действие.

### 6.4. Backend

Нужен отдельный endpoint:

- `GET /api/v1/products/{product}/movements`

Пример ответа:

- `current_stock`
- `movements[]`
  - `type`
  - `quantity`
  - `price`
  - `total`
  - `created_at`
  - `reference_id`
  - `reference_type`
  - `actor_name`

Источники движения:

- `purchase_items`
- `sale_items`

## 7. Раздел «Продажи»

Продажи должны оформляться максимально быстро.

### 7.1. Экран списка продаж

Должен содержать:

- поиск по клиенту;
- фильтр по дате;
- список продаж.

Каждая запись должна показывать:

- ID или номер;
- имя клиента;
- сумму;
- оплачено;
- долг;
- дату.

### 7.2. Экран создания продажи

Должен содержать:

- поиск товара;
- добавление товара в корзину;
- количество;
- цену;
- скидку;
- оплачено;
- тип оплаты:
  - `cash`
  - `card`
  - `transfer`
- итог;
- автоматически рассчитанный долг, если оплачено не полностью.

### 7.3. Экран детализации продажи

Должен содержать:

- клиент;
- список товаров;
- количество;
- цены;
- скидку;
- оплачено;
- долг;
- дату;
- автора продажи.

## 8. Раздел «Долги»

Один из центральных разделов приложения.

### 8.1. Экран списка долгов

Должен содержать:

- поиск по имени;
- фильтр:
  - все;
  - активные;
  - закрытые;
- список долгов.

Каждая запись должна показывать:

- имя;
- баланс;
- последнюю операцию;
- дату.

### 8.2. Экран карточки долга

Должен содержать:

- имя;
- текущий баланс;
- историю транзакций;
- действия:
  - `Добавить долг`
  - `Погашение`
  - `Коррекция`

### 8.3. Логика статусов

Нужно отображать:

- `есть долг`;
- `закрыт`.

### 8.4. Логика операций

- `give` увеличивает долг;
- `repay` погашает долг;
- `take` уменьшает долг через возврат или обратную операцию.

## 9. Раздел «Расходы»

Этот раздел особенно важен для `owner`.

### 9.1. Экран списка расходов

Должен содержать:

- поиск;
- фильтр по дате;
- список расходов.

Каждая запись должна показывать:

- название;
- количество;
- цену;
- итог;
- дату.

### 9.2. Экран создания расхода

Поля:

- название;
- количество;
- цена;
- примечание.

### 9.3. Рекомендация на будущее

Позже желательно добавить категории расходов:

- аренда;
- доставка;
- коммунальные;
- зарплата;
- прочее.

## 10. Раздел «Закупки»

Это owner-раздел для прихода товара.

### 10.1. Экран списка закупок

Должен содержать:

- поставщика;
- сумму;
- дату;
- количество позиций.

### 10.2. Экран создания закупки

Поля:

- поставщик;
- список товаров;
- количество;
- цена;
- итог.

### 10.3. Важная логика

После закупки остаток товара должен обновляться автоматически.

## 11. Раздел «Отчёты»

На мобильной версии отчёты должны быть лёгкими и понятными.

### 11.1. Минимальный состав

- продажи;
- расходы;
- прибыль;
- остатки;
- low stock;
- долги.

### 11.2. Фильтры

- сегодня;
- 7 дней;
- 30 дней;
- свой период.

### 11.3. Формат

Лучше использовать:

- KPI-карточки;
- один простой график;
- один список проблемных точек.

Не рекомендуется перегружать мобильный интерфейс сложной BI-аналитикой.

## 12. Раздел «Ещё»

Должен содержать:

- профиль;
- настройки;
- пользователи;
- валюта;
- магазин;
- выход.

Для `seller` список должен быть короче и без owner/admin-функций.

## 13. UI/UX требования

### 13.1. Общие принципы

Интерфейс должен быть:

- быстрым;
- понятным;
- минималистичным;
- ориентированным на частые ежедневные действия.

### 13.2. Правила

- минимум вложенности;
- минимум тяжёлых таблиц;
- максимум карточек и крупных CTA-кнопок;
- важные цифры должны быть видны сразу;
- на главной только ключевая информация;
- фотографии товаров должны использоваться в списке и карточке товара;
- продажа должна оформляться за минимальное количество шагов.

## 14. Что уже есть в backend

В текущем backend уже есть основа для:

- аутентификации;
- магазинов;
- пользователей;
- товаров;
- продаж;
- закупок;
- расходов;
- долгов;
- отчётов;
- загрузки картинки товара.

## 15. Что желательно добавить в backend

Для удобного mobile UX рекомендуется добавить:

- `GET /api/v1/dashboard`
- `GET /api/v1/products/{product}/movements`
- `GET /api/v1/products/low-stock` или аналог через фильтр
- `GET /api/v1/sales/recent` или включить в dashboard
- `GET /api/v1/debts/summary` или включить в dashboard

### Этап 2

- расходы;

## 17. Итоговая структура MVP

Для первой полноценной версии рекомендуется следующий состав экранов:

- логин;
- главная;
- товары;
- карточка товара;
- движение товара;
- продажи;
- долги;
- расходы;
- раздел `Ещё`.

Это покроет основной ежедневный цикл работы магазина и будет соответствовать ожиданиям клиента по присланным референсам.
